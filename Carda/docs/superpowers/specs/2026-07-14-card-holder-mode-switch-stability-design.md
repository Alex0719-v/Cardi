# 名片夹快速切换与滚动稳定性修复设计

## 目标

修复“名片夹”在快速切换“列表 / 姓名 / 公司”并快速上下滚动时，偶发组件错位、分组文字或其他文字消失的问题，同时完整保留当前 Figma 视觉、布局和动画表现。

## 不可变约束

- 不修改任何 Figma 坐标、尺寸、间距、颜色、圆角、层级或材质。
- 不修改任何现有动画曲线、时长、延迟、transition、opacity、offset、缩放或 Union 形变。
- 不通过节流、延迟点击、禁用按钮、降低滚动速度或等待动画结束来规避问题。
- 不改变列表、姓名、公司三种模式现有的滚动位置语义、展开状态、排序结果或交互入口。
- 不扩大到无关重构；修复仅处理旧交互回调和旧几何结果污染当前模式的问题。

## 根因

### 1. 旧滚动手势可以在模式切换后继续写入共享顶部状态

`ScrollViewPanObserver.Coordinator` 在现有 `UIScrollView.panGestureRecognizer` 上追加 target。协调器在 SwiftUI 更新时会替换 `onChanged` / `onEnded` 闭包，但没有记录手势开始时所属的模式或交互代际。

当一个 pan 已经在旧模式开始、用户随即切换模式时，迟到的 `.changed` 或 `.ended` 事件会调用更新后的闭包。此时事件在物理上属于旧模式，逻辑上却被当成新模式事件处理，可能在 `selectMode` 已经把顶部重置为展开态后再次写入收起方向或清除新模式的方向状态。共享的 `CardHolderHeaderCollapseState` 同时驱动背景 Union、顶部栏、滚动模块和遮罩，因此一次过期写入会让不同组件在快速事务中出现错位。

### 2. 分组标题锚点缺少模式身份且偏好归并只追加不去重

姓名和公司模式都使用字母作为 `GroupedCardSection.id`，而 `GroupedHeaderAnchor` 只保存 `title` 与 `bounds`。模式动画交叠或 LazyVStack 回收/重建期间，偏好归并可能同时得到旧模式与当前模式的同名字母锚点。

`GroupedHeaderAnchorPreferenceKey.reduce` 当前直接追加，前景 `ForEach` 又使用字母标题作为身份。重复身份和旧 frame 会让 SwiftUI 无法稳定判断应绘制哪一个标题。由于滚动内容中的真实分组标题固定为透明，前景锚点一旦为空、重复或取到旧几何，用户看到的结果就是标题消失或位置错误。

## 修复设计

### 手势会话代际隔离

- `CardHolderView` 维护一个仅用于正确性校验的模式交互代际值。
- 每次真正切换模式前先递增代际，再沿用现有代码重置顶部状态并播放原有模式动画。
- `ScrollViewPanObserver.Coordinator` 在 pan `.began` 时锁定当时代际。
- `.changed` 和 `.ended` 只有在“手势开始代际 == 当前代际”时才允许向 `CardHolderView` 回调。
- 模式切换前已开始的旧手势仍由系统正常结束，但其后续事件只做内部清理，不再修改新模式状态。
- 观察器继续附着在系统现有 pan recognizer 上，不新增竞争手势，不改变滚动物理。

### 分组标题锚点模式隔离

- 为每个锚点增加 `HolderMode`，稳定身份改为“模式 + 标题”。
- 偏好归并按稳定身份去重；同一布局周期出现重复值时保留最新锚点，避免重复 `ForEach` 身份。
- 前景绘制、吸顶标题解析、透明度计算和 retention 回写只消费当前模式锚点。
- 旧模式在动画生命周期内仍可完成原有淡出，但其锚点不能参与当前模式的 frame 计算或 retention 状态。
- 现有分组标题 transition、delay、opacity 计算、吸顶位置和遮罩保持不变。

### 状态写入边界

- `headerCollapseState.offset` 的 0 / 110pt 规则、0.20 秒动画和 cubic-bezier 参数保持不变。
- `mode` 仍按当前 `.snappy(duration: 0.28)` 流程更新。
- `groupedScrollPosition`、`listScrollPosition` 以及程序化滚动逻辑保持不变。
- 修复不引入模式切换锁、时间 suppression、debounce 或新的异步延迟。

## 数据流

1. 系统 pan 开始，观察器记录当前模式交互代际。
2. 手势逐帧变化；代际未变化时按现有方向规则更新顶部二态。
3. 用户切换模式，`CardHolderView` 递增代际、清除方向跟踪并按现有逻辑展开顶部和更新模式。
4. 旧 pan 的迟到事件因代际不匹配被丢弃；新模式首次新 pan 开始后记录新代际并正常工作。
5. LazyVStack 上报带模式身份的标题锚点；前景层仅解析当前模式的去重结果。

## 测试设计

### 自动回归

- 扩展现有 `CardHolderScrollStabilityUITests`，加入三模式快速轮换与快滑交叠场景。
- 每轮覆盖姓名 → 公司 → 列表 → 姓名，并穿插快速上滑、下滑。
- 最终主动展开顶部后验证：`名片夹`、`列表`、`姓名`、`公司` 均存在，模式按钮可点击且 frame 位于画板范围内。
- 姓名和公司模式分别验证至少一个分组标题及一个折叠名片文字存在、frame 非空且与画板相交。
- 失败时保留截图，便于区分顶部位移错误、分组标题丢失和名片内容丢失。
- 原有列表模式慢速/连续快速方向切换和跨页面返回测试必须继续通过。

### 实现级验证

- 为代际门控建立可重复的测试：旧代际手势在模式切换后产生 `.changed` / `.ended` 时不得触发业务回调；新代际手势正常触发。
- 为锚点归并建立可重复的测试：相同“模式 + 标题”只保留一个结果；旧模式同名字母不能进入当前模式集合。

### 零视觉/动画改动审计

- 对实施前后的相关 diff 做静态审计。
- `Animation`、`withAnimation`、`timingCurve`、`snappy`、`transition`、`opacity`、`offset` 以及所有 Figma 常量不得发生修改。
- 使用修复前后的正常速度截图检查列表、姓名、公司三态；像素结构和动画参数必须保持当前实现。

## 验收标准

- 快速切换三种模式并快速上下滚动后，不出现组件错位或任何文字消失。
- 旧模式的 pan 或标题锚点不能修改当前模式状态。
- 当前视觉、布局、动画曲线、时长和过渡与修复前一致。
- 定向实现级测试、名片夹 UI 回归测试及应用构建全部通过。
