## 2. 核心产品哲学

### 2.1 名片优先

名片是 Cardi 的核心对象。所有页面、交互、数据模型和视觉系统都围绕名片展开。

### 2.2 创建、交换、管理三主线

- **创建**：用户可以创建一张或多张自己的名片。
- **交换**：用户在“我的名片”页面上划当前名片，主动递给 1.5 米内的 Cardi 用户；设备支持方向测量时选择方向明确且最近的人，只支持精准距离时仅选择唯一且距离明确最近的人。双方在短时间内上划则自动互换，单向接收后可翻转名片回递。当前实现不得调用 AirDrop、NameDrop、二维码或手机碰手机 NFC。
- **管理**：用户在名片夹中查看、分类、搜索、复制、分享和保存收到的名片。

### 2.3 视觉与交互要贴近 Figma

当前 UI 以 Figma Page 1 为唯一视觉参考。实现时应优先还原 frame 的结构、尺寸、间距、圆角、玻璃质感和动效关系。

全局背景规则：
- 默认页面背景使用 `PageBackground` / `#F9F8FB`。
- 例外：创建/编辑名片、“我的名片”、名片夹大面积内容面板与搜索页面统一使用 `SearchBackground` / `#D8D7DB`；`MyCardsBackground` 与 `EditorBackground` 均为该 token 的别名。
- 新版“名片夹”页面的面板轮廓仍以 Figma `10064:3390` / `10064:3656` 为准，但不再使用白色叠加 `#C3C2C8` 的 50% / 65% 复合灰，统一填充 `SearchBackground`。
- Cardi 最低支持 iOS 17.0，且当前为固定浅色视觉系统。应用根视图必须使用 `.preferredColorScheme(.light)`；不得让系统深色模式把 Liquid Glass 或渐变毛玻璃切换为深灰/黑色材质配方。
- `AppShellView` 根背景与页面内容动画必须分层。主页面内容继续使用 0.24 秒等时长交叉淡化：旧页面内容不透明度降低、新页面内容不透明度增加。背景不得做 opacity 渐变：“我的名片”与搜索/最近添加都对应铺满屏幕的 `SearchBackground` / `#D8D7DB` 状态；它们进入名片夹时，同一块灰色区域顶部从 y=0 向下移动到 y=126，并形变为名片夹当前列表/姓名/公司模式对应的内容面板与选中标签 Union；从名片夹返回“我的名片”或进入搜索/最近添加时沿原路径向上移动并重新铺满页面。动画下方固定显示名片夹的白色叠加 `#303136` 6% 底色。不得在整个 AppShell 上扩散隐式动画。

全局字体规则：
- 软件中所有中文文本使用苹方 / PingFang SC。
- 软件中所有英文、拼音、电话、邮箱、链接等拉丁字符信息使用 SF Pro。
- 不使用乱码字符或 Figma 字体码位代替真实图标或文本。

### 2.4 不确定即确认

如果需求、视觉、交互、数据结构或系统能力存在不明确处，必须先询问用户，不得自行猜测。

### 2.5 代码与文档同步

开发时必须同时维护本地 `.md` 指导文件。凡是用户确认了新需求、设计约束、资源来源、实现方式或当前限制，相关代码完成时必须同步更新对应文档。

同步范围：
- `AGENTS.md`：项目级长期规则、不可违反约束、关键产品逻辑。
- `contributing_ai.md`：AI 协作流程、验证要求、变更协议。
- `README.md`：当前项目状态、运行方式、目录结构、已接入资源。
- `Designsystem.md`：视觉 token、Figma 坐标、图标/玻璃/字体/布局规则。
- `ROADMAP.md`：功能完成度、进行中事项、阻塞项和技术债。

如果某次代码改动只影响其中部分文档，应只更新相关文档；但不得让文档继续保留与实现冲突的旧描述。

## 3. 核心页面结构

Cardi 主要有三个页面：

1. **我的名片**
2. **名片夹**
3. **名片搜索**

底部导航由左侧胶囊型导航与右侧搜索圆形按钮组成。搜索被激活时，右侧圆形按钮展开成横向搜索栏，左侧胶囊导航收缩成一个返回当前页面的按钮。

底部导航材质统一使用原生 Liquid Glass，并以 Figma `10063:3134` 为轮廓与图层来源。Figma 原始三层填充为 65% 白色 Normal、`#DDDDDD` Color Burn、`#F7F7F7` Darken；按最新增强指令，当前实现将三层可见强度分别调整为 44%、48%、42%，减少乳白覆盖，让原生折射更明显；玻璃基底仍为 0.4% 黑色，阴影增强为 x=0/y=10/blur=44/spread=0/16% 黑色。Figma GLASS 原始参数为 radius=7、refraction=1、depth=16、lightAngle=315°、lightIntensity=0.8、dispersion=0、splay=0.06、cornerSmoothing=0.6；没有普通 stroke，液态玻璃描边来自 GLASS 高光折射。iOS 26 及以上必须使用不带额外 tint 的原生 `.glassEffect(.regular)`，左侧导航、搜索按钮/搜索栏与清空按钮放在同一个 `GlassEffectContainer(spacing: 0)` 中，防止进入搜索页时相邻表面因距离自动吸引、融合或形变；可点击表面继续追加 `.interactive()`，点击反馈只能由真实按压触发。其他公共玻璃按钮同样取消额外白色 tint，以扩大真实玻璃的取色与折射范围。选中 tab 保留不透明 `#EDEDED` 灰色滑块并绘制在玻璃之上。低于 iOS 26 时不得引用 Liquid Glass API，必须使用浅色 `ultraThinMaterial` / 毛玻璃降级，并继续受应用固定浅色模式约束。

底部导航在“我的名片 / 名片夹”真正发生页面切换时必须提供克制的同步反馈：新选中的 SVG 图标先在 0.08 秒内缩小到 90% 并下移 0.8pt，再以低 bounce 弹簧放大到 104.5%、上移 0.4pt，最后回到 100%；整条底栏同期轻微压缩到 x=99.6% / y=97.8% 并下移 0.6pt，再回弹到 x=100.2% / y=100.6% 后归位。灰色选中滑块单独参考登录/注册页原生 segmented control：使用 0.24 秒、零额外回弹的 `.smooth` 位移动画，不再使用 `.snappy`；开启“减弱动态效果”时滑块立即到位。重复点击当前 tab 不触发；进入搜索态的现有 0.68 秒形变不使用这套反馈。开启“减弱动态效果”时禁用缩放回弹。

底部导航进入搜索态时不得通过条件替换生硬跳转：必须先保留 `morphingIdleControls` 完整播放形变，再于 0.68 秒结束后在相同几何位置无动画交接给 `searchActiveControls`。左侧 191 x 62 跑道圆连续收缩为 62 x 62 完全倒角正方形，右侧 62 x 62 搜索按钮连续拉伸为 279 x 62 跑道圆；左侧两个图标同时向收缩后圆心 x=51.5 移动，当前选中图标保留，未选中图标从收缩开始减淡，并在两图标视觉边界开始相交时完全透明。主形变统一使用奇幻形变的 `cubic-bezier(0.2, 0.72, 0.18, 1)`、0.68 秒曲线；未选中图标淡出使用同曲线的前 0.42 秒，反向展开时延后 0.26 秒出现。退出搜索页时按相反顺序交接并播放反向形变。

底部导航左侧两个 tab 的图标来源为项目本地 `My icon/` 文件夹：
- `mycard-1.svg`：我的名片未选中
- `mycard-2.svg`：我的名片选中
- `cards-1.svg`：名片夹未选中
- `cards-2.svg`：名片夹选中

实现要求：
- 不再自行手绘“我的名片”和“名片夹”导航图标。
- 不使用乱码字符或 Figma 字体码位代替图标。
- 底部导航应直接读取 `My icon/` 中的 SVG 资源。
- 高频页面切换中的 SVG 图标必须使用本地缓存的 SwiftUI 渲染路径，不得用 `WKWebView` / HTML 异步加载图标，避免图标闪烁或延迟出现。

所有主页面右上角的玻璃质感圆圈表示当前用户头像：
- “我的名片”与名片夹姓名/公司模式的账户头像统一为 x=342、y=62、w=44、h=44，不得在任一页面额外增加顶部偏移。
- “我的名片”的静态 `UserAvatarButton` 与名片夹的动态头像必须复用同一个 `AccountAvatarGlassSurface`：44 x 44 终态使用 `.circular` `RoundedRectangle`、cornerRadius=22，玻璃图形先锁定 frame 再应用 `.glassEffect`；登录头像图片通过 overlay 以 `scaledToFill` 填满并裁切到同一轮廓，不得退回旧 `FigmaGlassShape + Circle` 组合。
- 未登录时：显示 Figma 中的空白玻璃样式。
- 登录后：显示用户设置的头像。
- 右上角头像必须来自登录账户头像数据源，不得读取当前名片或任意名片上的头像。
- 在“我的名片”已有名片时，点击右上角头像会打开底部弹窗，可继续添加名片。

## 4. 我的名片

当前新版 Figma 节点：`10063:3262`

页面背景：
- 使用与名片搜索页相同的 `SearchBackground` / `#D8D7DB`。
- `MyCardsBackground` 作为 `SearchBackground` 的别名保留，不再使用旧值 `#E1E0E3`。
- “我的名片”页面中的完整名片不绘制名片本体外框阴影；名片右侧操作按钮也不绘制阴影。

### 4.1 空状态

Figma 节点：`4268:13366`

首次进入且没有任何名片时，页面展示一张居中的灰色空卡片。点击中间灰色卡片进入创建/编辑名片界面。

空状态当前 Figma 尺寸：
- 画板：402 x 874
- 空卡片：x=16, y=158, w=370, h=223
- 加号横线：x=185, y=268, w=32, h=4
- 加号竖线：x=199, y=285, w=4, h=32
- 空状态加号必须使用单个复合路径一次描边，再对整个图形统一应用 75% 透明度；不得叠加两条半透明图形，避免交汇处颜色变深。
- 底部导航栏：x=0, y=779, w=402, h=95
- 左侧胶囊导航：x=25, y=16, w=183, h=54；左边缘不变，右边缘累计向左收回 1pt
- 右侧搜索圆球：x=323, y=16, w=54, h=54

### 4.2 创建/编辑名片

Figma 节点：`10599:4144`

页面配色：
- 页面底色使用 `#D8D7DB`。
- 公司/职位/LOGO 与动态字段两组输入面板使用不透明 `#F3F3F3`。
- 未上传头像时，头像编辑入口显示 `Assets.xcassets/EditorDefaultAvatar.imageset/Light.svg`，不得再使用纯色圆或系统头像图标替代。
- 新增字段按钮使用白色圆底，加号使用 `#D8D7DB`。
- 删除字段按钮继续使用 `#FF1212`，减号与表单分隔线继续使用 `#E6E6E6`。

页面行为：
- 左上角返回按钮只回到进入创建/编辑页之前的页面，不创建也不修改名片。
- 右上角确认按钮才会提交创建或修改。
- 顶部卡片展示当前名片样式预览。
- 顶部预览窗口提供三张本地名片底图：`Card photo/Group 42.png`、`Card photo/color2.png`、`Card photo/color3.png`；创建新名片默认使用第一张，编辑已有名片时从该名片当前底图开始。
- 用户在顶部预览区域左右滑动即可按固定顺序切换底图，首尾停止。预览页步长固定为画板宽 402pt（370pt 名片宽 + 32pt 间距），相邻底图与右侧圆形按钮底色从页面最侧边进入，曲线和时长复用“我的名片”分页；名片上的文字、头像、公司 Logo 与按钮图标保持在原坐标不移动。名片正下方显示 3 个直径 8pt、间距 12pt 的活动圆点，不绘制外层跑道圆。切换只更新编辑草稿，点击右上角确认后才持久化；左上角返回仍放弃本次底图修改。
- 圆形区域为名片头像上传位置。
- 如果未上传头像，最终名片中不显示头像圆形。
- 姓名可点击输入或修改。
- 姓名下方的英文名/拼音可点击输入或修改。
- 灰色信息框中的灰色文本均可输入或修改。
- 未输入内容时，编辑页必须显示 Figma 默认占位文本，不得显示开发示例文本。
- 姓名默认占位为 `姓        名   `，并保持大字距；“名”字在此前向左移动 36pt 的基础上继续向左移动 7pt，当前总偏移为 43pt。英文名/拼音默认占位为 `Xing Ming`，仅占位文本使用 Figma 16px 字距；用户输入后必须恢复正常字距。
- 姓名默认占位必须完整显示为 `姓        名   `，不得截断为 `姓…`。
- 姓名和英文名/拼音输入栏为固定单行输入，不允许上下拖动或内部竖向滚动。
- 姓名或英文名/拼音为空时，进入对应输入框的编辑状态后，占位文字透明度降为 40%；开始输入后占位消失，离开空输入框后恢复为 100%。
- 创建/编辑页所有文本输入键盘的右下角使用确认/完成键，不显示换行键；点击后只结束当前输入并主动收起键盘，不提交整张名片。
- 手机、邮箱、地址和链接字段虽然允许按宽度自动折行，但必须拦截键盘确认产生的换行字符：确认时保持原文本并收起键盘，不得插入手动换行。
- 手机号输入与名片展示统一按 `3 4 4` 自动分组；例如输入 `19204902910` 时实时显示并保存为 `192 0490 2910`。拨号和搜索必须忽略这些显示空格。
- 编辑手机号、邮箱、地址或链接时，无论字段为空还是已有内容，每次进入输入/编辑状态都必须重新触发滚动，将当前输入行抬到系统键盘上方的统一固定位置；结束输入后恢复到进入该字段前的滚动位置。连续切换字段时不得先回弹再抬起。
- 每个文本外框高度需要根据输入文本内容自适应。
- 黑色字段类型文本右侧带小箭头，点击后不是直接输入，而是选择类型。
- 字段类型选项：手机、邮箱、地址、链接。
- 公司 LOGO 不属于动态字段类型，只能通过公司/职位信息组中的固定 `LOGO` 行上传。
- 右侧红色圆形减号用于删除对应文本框。
- Figma 默认四个动态字段中，首行手机不显示删除圆圈，邮箱/地址/链接三行显示删除圆圈。
- 底部加号用于新增文本框。
- 动态信息栏必须根据文本自适应行高：基础行高 68；新增 1 栏总高度增加 68，删除 1 栏总高度减少 68；任一字段文本每多换 1 行，该小单元和总高度都额外增加 20。

编辑页当前 Figma 尺寸：
- 画板：402 x 1044
- 顶部预览名片：x=16, y=63, w=370, h=222, cornerRadius=24
- 头像编辑入口：x=43, y=240, w=87, h=90
- 姓名：x=16, y=340, w=210, h=41
- 英文名/拼音：x=16, y=385, w=247, h=22
- 公司/职位信息组：x=16, y=425, w=370
- 公司/职位/LOGO 信息组必须作为独立固定组件重新定位文字，不得复用动态信息栏的字段行组件
- 公司/职位信息组：h=204，三行各 68；公司行 y=0、职位行 y=68、LOGO 行 y=136，避免被分隔线、图片选择器或自动布局挤偏
- 公司与职位行的标题位置固定为 x=16 y=13，输入/占位文本固定为 x=16 y=35，文字宽度 338
- LOGO 行必须先绘制可见文字与箭头，再覆盖透明图片选择点击层，避免 LOGO 行内容被 PhotosPicker 布局吞掉

- 动态字段行：基础 h=68，换行每多 1 行增加 20；每行 y 坐标由前序行高度累加得到

### 5.7 名片动态高度规则

基准状态：
- 最多 3 段单行信息时，名片高度为 222pt。

动态增长：
- 如果信息字段超过 3 段，每多 1 段，名片高度增加 24pt。
- 如果最后一条以外的某一段信息发生换行，每多 1 行，名片高度额外增加 20pt；最底部最后一条固定单行，不增加换行高度。
- 信息字段文本单元本身也必须按每多 1 行增加 20pt 排布，确保总高度增长后信息模块仍按底部锚定。
- 横向宽度 370pt 保持不变。
- 名片左上角在页面中的 x/y 不应因高度变化而漂移，除非后续用户明确要求。
- 公司组与头像保持顶部锁定。
- 姓名组保持竖向居中。
- 信息模块保持底部锁定。
- 右侧操作按钮列保持底部锁定；动态高度公式不因按钮列额外增加高度。

建议公式：

```text
cardHeight = 222
  + max(0, visibleInfoFieldCount - 3) * 24
  + wrappedExtraLineCount * 20
```

其中：
- `visibleInfoFieldCount` 为右下角实际显示的信息字段数量。
- `wrappedExtraLineCount` 为除最底部最后一条外，其余字段换行后超出第一行的总行数。

### 5.8 保存为图片

保存名片为图片时：
- 输出格式为 `.png`。
- 输出内容只包含名片本体，不包含页面背景、导航栏、弹窗或遮罩。
- 输出应使用当前名片真实动态高度。
- 输出轮廓固定为完整直角矩形：去除屏幕展示所用的外圆角、右侧按钮圆形底图和右下角按钮槽裁切。
- 头像、公司 logo、底图、信息文字和右侧字段图标仍必须被渲染进图片；字段图标仅在导出中缩小到屏幕尺寸的 75% 并整体向左移动 10pt。每个导出图标必须根据前序信息行的真实高度和 6pt 行间距自动定位，竖向居中对齐对应信息的第一行；右对齐信息文字列相对屏幕名片总计向右移动 15pt，即在上一版导出位置上继续右移 10pt。
- 以上导出专用几何不得影响“我的名片”、名片夹、搜索、编辑预览或交换动画中的屏幕名片。

## 8. 名片交换

Cardi 的主要交换方式为“上滑交换”，不调用 AirDrop、NameDrop、二维码或手机碰手机 NFC。iOS 的 NameDrop / Bringing Devices Together 是系统级手势，第三方 App 不能拦截或关闭；Cardi 不要求两台手机顶部贴合，而是把用户上划作为明确发送意图，再用 Nearby Interaction 确认目标距离，并在设备支持时同时确认方向。

交换对象永远是“我的名片”页面当前正在展示的那张名片。

### 8.1 当前启用边界

- 只有双方都打开 Cardi、App 处于前台，并停留在“我的名片”页面时才启用前台交换能力。
- 当前页面必须存在可交换的当前名片。
- 创建/编辑页、添加名片弹窗或离开“我的名片”页面时，必须停止 Multipeer Connectivity 与 Nearby Interaction session。
- 页面前台待机时只启动 `MCNearbyServiceAdvertiser`，作为低功耗可接收状态；不得同时长期运行 browser 或 UWB 测距。
- 用户从当前名片内开始向上拖动，位移达到 20pt 后才启动 `MCNearbyServiceBrowser`、连接和短时 Nearby Interaction 测距；上划达到 120pt 或预测位移达到提交阈值后才提交递卡意图。
- 递卡 `DragGesture` 必须挂在当前名片轮播的稳定命中面，并在“我的名片”画板命名坐标系中校验起点；不得与整页分页手势并列挂在页面祖先层，否则点击、长按和横向分页的组合识别会使部分真机完全收不到递卡手势。交换修改必须保留一条从屏幕名片内真实合成上划、并断言已进入 NI 能力检查或主动发现状态的 UI 回归。
- 主动发现窗口为 10 秒；取消、失败、完成、超时、离开页面或 App 进入后台后停止 browser 与测距并清理连接，保留前台 advertiser 待接收。
- 收到的名片保存为 `ownerKind = .received`，进入名片夹数据源。
- 用户上划 20pt 时必须先复制当前完整名片 UI 形成独立动画层；原页面名片在手势跟随、等待目标、递出形变和动画结束的全过程中保持位置、透明度与缩放不变。动画副本从 0.96 倍开始实时跟随 `DragGesture.translation`，随上划距离连续增长到最高 1.08 倍，并按预测速度加入不超过 5° 的轻微 x 轴姿态；目标锁定时用低幅度弹簧额外向递出方向吸附 12pt，并至少保持 1.05 倍。取消手势时所有状态可中断并用快速收敛弹簧回到原位。提交时若目标尚未锁定，副本同样立即回到原位并移除，后台的短时发现继续进行；之后锁定目标时从原卡位置创建递出动画，禁止把副本冻结在手势终点等待超时。提交时已经锁定目标才沿用同一个稳定动画 ID 进入递出。
- 发送阶段继续使用上述同 ID 动画实体，从手势结束位置连续进入两段式奇幻形变：前 0.24 秒从当前缩放收至约 0.85 倍并轻微倾斜，随后约 0.85 秒使用 SwiftUI 原生 `distortionEffect` + `Components/CardGenieShader.metal` 将整张名片作为一块连续弹性表面沿短弧线向屏幕上方目标收束到约 15% 宽度，并只在最后 22% 进度内淡出。禁止重新使用独立横向切片、突然替换视图或简单直线缩放；形变过程中名片各行必须连续相接，不得出现层叠、缝隙或多张卡错觉。递出动画完成后停顿 0.1 秒，再播放排队的接收名片动画。
- 接收名片动画以 Figma `10050:2742` 为准：接收态叠加全屏背景模糊和 `rgba(158,158,158,0.5)` 灰色遮罩，左右上角显示“拒绝”和“分到列表”按钮；来卡不得再使用普通直线滑入，而要复用同一 `distortionEffect` 奇幻形变的反向过程，从屏幕顶部 x=屏幕中心、y=-18 的 15% 宽度收束态出发，在约 0.85 秒内沿短弧线连续展开并恢复为 x=16、y=326、w=370 的完整名片。该动画结束后仍由同一视图承载侧向翻转手势，不得通过淡入新卡替换。3 秒内未点击“拒绝”则自动接收。普通收进名片夹时再正向运行同一形变：靠近名片夹按钮的一侧先被吸引，整张名片保持连续表面并沿弯曲路径逐步收窄、汇聚，在约 1.2 秒内于 x=163.5、y=822 处淡出；不得退回水平分片堆叠实现。动画结束后才写入名片夹。
- 点击“分到列表”不得立即收纳；必须先显示现有 `BusinessCardList` 的单选弹窗。用户选中一个列表并点击“确认”后，才播放原有收纳动画、把收到的名片写入该列表并发送 `persistedAck`；重复名片使用覆盖策略时，明确选择的列表同时覆盖原归属。弹窗显示 `取消` 按钮，点击该按钮或弹窗外遮罩均可关闭并返回接收界面。弹窗打开或用户正在翻转名片时，3 秒自动接收计时必须暂停并重置；关闭弹窗后重新完整计时。没有现有列表时显示空状态并禁用确认。
- 单向 `delivery` 来卡允许侧向翻转回递。翻转使用约 0.5 秒受控的横向压缩—换面—展开，不让 3D 合成层扩散到名片边界外。翻面完成后必须先保存来卡并建立 `persistedAck` / 新 `returnDelivery` 链路；随后只把翻面后出现的本机当前名片从原位置交接给上层连续奇幻形变并向上递出，释放阶段约 0.14 秒、形变阶段约 1.05 秒。此场景不得再同时播放对方名片收进名片夹的动画，避免双动线抢夺主视觉；普通接收、自动接收和“分到列表”仍保留约 1.2 秒奇幻收纳。互换与回递来卡不得再次触发翻面。交换动画中的名片和名片上的右侧圆形按钮不绘制阴影。
- 奇幻形变依赖 Xcode 官方可选 Metal Toolchain，不引入第三方运行库；构建环境缺少该组件时使用 `xcodebuild -downloadComponent MetalToolchain` 安装。开启“减弱动态效果”时必须自动降级为短距离移动、缩放与淡出，不执行大幅弹性扭曲。

### 8.2 目标选择

- Nearby Interaction 支持性以 `supportsPreciseDistanceMeasurement` 为准，不得把 `supportsDirectionMeasurement == false` 误判为整台设备不支持交换。
- 设备支持方向测量时，候选人必须满足距离不超过 1.5 米、`NINearbyObject.direction` 位于手机前方约 ±45°、读数连续稳定至少 0.4 秒。
- 设备只支持精准距离时，允许使用 distance-only 降级：候选人仍须在 1.5 米内且稳定至少 0.4 秒，并且只能选择唯一、距离明确最近的设备；不得伪造方向或显示“已对准”。
- 距离使用最近 5 次有效读数的中位数，读数超过 1.25 秒视为过期。
- 多人场景按有效距离从近到远排序；最近两人距离差小于 0.3 米时视为目标不明确，不自动发送。方向模式提示用户将手机朝向对方；distance-only 模式提示用户拉开与其他附近设备的距离。
- 方向模式下 direction 缺失或目标位于前方锥体之外不得发送；两种模式下距离超出范围、读数不稳定或最近目标存在歧义均不得发送。
- 一个 peer 使用一个独立 `NISession`；连接后双方交换 discovery token，再用 `NINearbyPeerConfiguration` 启动测距。

### 8.3 单向递卡、双方互换与回递

- 单方上划提交后，向明确目标发送 `throwIntent`；等待 1.5 秒双向意图窗口，未收到对方匹配意图时按单向递卡发送。
- 双方上划意图时间差不超过 1.5 秒且互相锁定时，双方都以 `mutual` 模式发送当前名片，自动升级为双向互换。
- 单向接收者翻转来卡时，必须先保存收到的名片并发送其持久化 ACK，再通过同一连接以新的 exchange ID 回递自己的当前名片。
- 同一发送手势只绑定一个 peer；同一 peer 的主动上划递卡保留 5 秒冷却，回递不受该冷却阻断。
- `NINearbyObject` 因 timeout 被移除时，必须自动重新 `run` 当前 `NISession`；`NISession` 意外 invalidated 时，必须丢弃旧 session、创建新 session、重新发送本机 discovery token，并用缓存的对方 token 恢复测距。该类系统测距中断不得作为常驻错误文案显示给用户。

### 8.4 数据校验、持久化 ACK 与反馈

- 收到名片 payload 后先执行大小、字段数量、字符串长度和字段顺序范围校验；校验失败不得展示、保存或 ACK。
- 网络层收到完整数据不代表成功，禁止立即发送普通 ACK。
- 只有接收动画结束、`BusinessCard(ownerKind: .received)` 写入 SwiftData 且 `modelContext.save()` 成功后，才向发送方发送 `persistedAck`。
- 发送方收到 `persistedAck` 后，才显示“对方已接住”并触发明确成功震动；网络发送完成或递出动画播放完成不得提前冒充持久化成功。双向互换使用两段紧凑触感，单向递卡使用成功触感。
- 接收方拒绝或保存失败时发送 `rejected`，发送方不得显示“已接住”。
- 发送 `card` 后必须启动 12 秒持久化 ACK 截止计时；收到 `persistedAck` / `rejected`、断线、显式发送失败或超时都必须取消并回收该事务。任何 `.error` 或 transport failure 都必须清除当前 `localIntent`、发送事务、主动发现和相关超时，确保下一次上划可重新开始；禁止只改错误文案而留下永久阻塞状态。

### 8.5 技术方案与权限

- 当前前台 MVP 继续使用 Multipeer Connectivity 建立点对点连接、发送协议消息与名片数据；service type 为 `carda-ex`，Bonjour service 为 `_carda-ex._tcp`。
- `Info.plist` 必须保留 `NSLocalNetworkUsageDescription`、`NSBonjourServices` 与 `NSNearbyInteractionUsageDescription`。
- Nearby Interaction 只负责距离及设备可用时的方向，不传输名片。Apple 文档规定支持精准距离但不支持即时方向的设备应继续使用 distance-only 会话，不能直接显示“不支持精准方向识别”并拒绝启动。
- iOS 26.5 SDK 已将部分 Multipeer Connectivity API 标为弃用并建议使用 Network framework；完成双真机 MVP 验证后应迁移传输层，业务状态机与消息协议不得写死在 Multipeer API 中。
- 当前不启用 Background Modes、Control Center / 锁屏 Control、Live Activity 或通知接收；后台接收需要另立技术验证任务，不得从前台 MVP 推断为已支持。

### 8.6 尚未实现 / 待验证

- 需要两台支持 Nearby Interaction 精确测距的 iPhone 做端到端真机验证；模拟器不能验证真实 UWB distance。
- DEBUG + Simulator 可使用 `CARDA_ENABLE_EXCHANGE_SIMULATION=1` 显示模拟按钮，并用 `CARDA_AUTO_SIMULATE_EXCHANGE=1` 自动触发模拟保存链路；追加 `CARDA_SIMULATE_SINGLE_DELIVERY=1` 时改为单向来卡，追加 `CARDA_AUTO_SIMULATE_RETURN=1` 时在来卡稳定后自动执行翻面、神奇效果收纳和原位回递。所有入口只用于动画、payload、SwiftData 持久化 ACK 与回递回归，不代表真实发现、方向或 UWB 测距验证，不得进入 Release 或真机正式体验。
- 一方未打开 App、后台接收、未安装 App 投递、Control Center / 锁屏 Control、Live Activity 与通知接收当前均未实现。
- 后续不得用 AirDrop、NameDrop 或 NFC 作为该近距离交换能力的实现方案，除非用户明确改变边界。
- 灵动岛 / Live Activity 交换 UI 当前暂停，工程不得保留 `CardaExchangeLiveActivity` Widget Extension、`ExchangeLiveActivityController`、`ExchangeLiveActivityAttributes` 或 `NSSupportsLiveActivities`。后续只有在用户明确恢复灵动岛需求时，才重新设计官方系统能力方案。

## 9. 当前代码结构

当前项目已建立第一版业务模块：

- `CardiApp.swift`：应用入口，配置 SwiftData `ModelContainer`。

- `Components/`：名片、折叠行、导航、头像、SVG 和渐变模糊等复用组件。

## 10. 数据模型方向

正式实现前需要建立名片数据模型。初步方向如下，写代码前仍需与用户确认字段命名和持久化策略。

核心对象：
- `BusinessCard`：一张名片。
- `CardField`：一段可变信息。
- `BusinessCardList`：名片夹中的一个自定义列表。
- `CardAsset`：头像、公司 logo 等用户上传图片资源；三张名片底图继续作为 App 内固定资源，不把 PNG 二进制复制进每张名片，但 `BusinessCard.backgroundTemplateRaw` 必须保存其稳定模板标识。

名片字段建议：
- `id`
- `ownerKind`：我的名片 / 收到的名片。
- `name`
- `phoneticName`
- `position`
- `organizationName`
- `backgroundTemplateRaw`：`color1` / `color2` / `color3`，缺失或未知值回退到 `color1`。
- `avatarImage`
- `companyLogoImage`
- `cardListID`：收到的名片当前唯一归属列表；为空或指向无效列表时进入 `未分类`。
- `fields`
- `createdAt`
- `updatedAt`
- `receivedAt`

信息字段建议：
- `id`
- `type`：手机 / 邮箱 / 地址 / 链接
- `value`
- `sortOrder`

公司 logo 使用 `companyLogoImage` / `companyLogoImageData` 存储，不再作为动态信息字段的可选类型。

列表字段：
- `id`
- `name`
- `sortOrder`
- `createdAt`
- `updatedAt`

## 11. AI 开发规则

AI 在修改 Cardi 代码或文档时必须：

1. 先阅读 `AGENTS.md`、`contributing_ai.md`、`README.md`、`Designsystem.md` 和 `ROADMAP.md`。
2. 以 Figma Page 1 与用户描述为主要依据。
3. 遇到任何不明确、不确定或存在多种实现方向的地方，先询问用户。
4. 不自行扩展用户未确认的页面或模板；当前底图选择只允许在编辑页顶部预览窗口内左右分页，不额外增加模板列表页或更多底图。
5. 不自行更改名片生成规则。
6. 修改名片布局、动态高度、保存图片或交换逻辑前，必须说明影响范围。
7. 不覆盖用户已有改动，不删除未经确认的内容。
8. 新增设计、架构、功能决策后，同步更新相关文档。
