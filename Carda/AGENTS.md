## 1. 产品定位

**项目名：** Carda

**产品类型：** 社交类名片软件。

**核心功能：**
- 名片创建
- 名片交换
- 名片管理

**一句话定位：** Carda 是一个用于创建、展示、交换与管理交互名片的 iOS 社交工具。

## 2. 核心产品哲学

### 2.1 名片优先

名片是 Carda 的核心对象。所有页面、交互、数据模型和视觉系统都围绕名片展开。

### 2.2 创建、交换、管理三主线

- **创建**：用户可以创建一张或多张自己的名片。
- **交换**：用户通过 App 内顶部碰一碰式近距离交互递交或交换当前展示名片；当前实现不得调用 AirDrop、NameDrop 或 NFC。
- **管理**：用户在名片夹中查看、分类、搜索、复制、分享和保存收到的名片。

### 2.3 视觉与交互要贴近 Figma

当前 UI 以 Figma Page 1 为唯一视觉参考。实现时应优先还原 frame 的结构、尺寸、间距、圆角、玻璃质感和动效关系。

全局背景规则：
- 默认页面背景使用 `PageBackground` / `#F9F8FB`。
- 例外：新版“我的名片”页面以 Figma `10063:3262` 为准，页面背景为 `MyCardsBackground` / `#E1E0E3`。
- 例外：新版“名片夹”页面以 Figma `10064:3390` / `10064:3656` 为准，页面底色使用白色叠加 `#303136` 6%，内容面板使用白色叠加 `#C3C2C8`（姓名/公司 50%，列表 65%）。
- Carda 当前为固定浅色视觉系统，应用根视图必须使用 `.preferredColorScheme(.light)`；不得让系统深色模式把 Liquid Glass 或渐变毛玻璃切换为深灰/黑色材质配方。

全局字体规则：
- 软件中所有中文文本使用苹方 / PingFang SC。
- 软件中所有英文、拼音、电话、邮箱、链接等拉丁字符信息使用 SF Pro。
- 不使用乱码字符或 Figma 字体码位代替真实图标或文本。

### 2.4 不确定即确认

如果需求、视觉、交互、数据结构或系统能力存在不明确处，必须先询问用户，不得自行猜测。

### 2.5 代码与文档同步

开发时必须同时维护本地 `.md` 指导文件。凡是用户确认了新需求、设计约束、资源来源、实现方式或当前限制，相关代码完成时必须同步更新对应文档。

同步范围：
- `AGENTS.md`：项目级长期规则、不可违反约束、关键产品逻辑。
- `contributing_ai.md`：AI 协作流程、验证要求、变更协议。
- `README.md`：当前项目状态、运行方式、目录结构、已接入资源。
- `Designsystem.md`：视觉 token、Figma 坐标、图标/玻璃/字体/布局规则。
- `ROADMAP.md`：功能完成度、进行中事项、阻塞项和技术债。

如果某次代码改动只影响其中部分文档，应只更新相关文档；但不得让文档继续保留与实现冲突的旧描述。

## 3. 核心页面结构

Carda 主要有三个页面：

1. **我的名片**
2. **名片夹**
3. **名片搜索**

底部导航由左侧胶囊型导航与右侧搜索圆形按钮组成。搜索被激活时，右侧圆形按钮展开成横向搜索栏，左侧胶囊导航收缩成一个返回当前页面的按钮。

底部导航左侧两个 tab 的图标来源为项目本地 `My icon/` 文件夹：
- `mycard-1.svg`：我的名片未选中
- `mycard-2.svg`：我的名片选中
- `cards-1.svg`：名片夹未选中
- `cards-2.svg`：名片夹选中

实现要求：
- 不再自行手绘“我的名片”和“名片夹”导航图标。
- 不使用乱码字符或 Figma 字体码位代替图标。
- 底部导航应直接读取 `My icon/` 中的 SVG 资源。
- 高频页面切换中的 SVG 图标必须使用本地缓存的 SwiftUI 渲染路径，不得用 `WKWebView` / HTML 异步加载图标，避免图标闪烁或延迟出现。

所有主页面右上角的玻璃质感圆圈表示当前用户头像：
- 未登录时：显示 Figma 中的空白玻璃样式。
- 登录后：显示用户设置的头像。
- 右上角头像必须来自登录账户头像数据源，不得读取当前名片或任意名片上的头像。
- 在“我的名片”已有名片时，点击右上角头像会打开底部弹窗，可继续添加名片。

### 4.2 创建/编辑名片

Figma 节点：`7575:14550`

页面行为：
- 左上角返回按钮只回到进入创建/编辑页之前的页面，不创建也不修改名片。
- 右上角确认按钮才会提交创建或修改。
- 顶部卡片展示当前名片样式预览。
- 当前顶部卡片固定展示 `Card photo/Group 42.png` 底图预览；底图切换页面尚未设计，不能自行补全。
- 圆形区域为名片头像上传位置。
- 如果未上传头像，最终名片中不显示头像圆形。
- 姓名可点击输入或修改。
- 姓名下方的英文名/拼音可点击输入或修改。
- 灰色信息框中的灰色文本均可输入或修改。
- 未输入内容时，编辑页必须显示 Figma 默认占位文本，不得显示开发示例文本。
- 姓名默认占位为 `姓        名   `，并保持大字距；英文名/拼音默认占位为 `Xing Ming`，Figma 字距为 16px。
- 姓名默认占位必须完整显示为 `姓        名   `，不得截断为 `姓…`。
- 姓名和英文名/拼音输入栏为固定单行输入，不允许上下拖动或内部竖向滚动。
- 创建/编辑页所有文本输入键盘的右下角使用确认/完成键，不显示换行键；点击后只结束当前输入并主动收起键盘，不提交整张名片。
- 每个文本外框高度需要根据输入文本内容自适应。
- 黑色字段类型文本右侧带小箭头，点击后不是直接输入，而是选择类型。
- 字段类型选项：手机、邮箱、地址、链接。
- 公司 LOGO 不属于动态字段类型，只能通过公司/职位信息组中的固定 `LOGO` 行上传。
- 右侧红色圆形减号用于删除对应文本框。
- Figma 默认四个动态字段中，首行手机不显示删除圆圈，邮箱/地址/链接三行显示删除圆圈。
- 底部加号用于新增文本框。
- 动态信息栏必须根据文本自适应行高：基础行高 68；新增 1 栏总高度增加 68，删除 1 栏总高度减少 68；任一字段文本每多换 1 行，该小单元和总高度都额外增加 20。

### 4.3 创建后的单张名片

Figma 节点：`9978:2322`（新版带右侧按钮名片）

创建成功后，“我的名片”页面展示真实名片。名片根据固定底图图片、头像、公司 logo、姓名、职位、英文名/拼音和信息字段动态生成。

名片底图规则：
- 使用项目 `Card photo/Group 42.png` 作为底图。
- 底图不重新制作，不用 SwiftUI 自绘 Figma 中的红橙光斑；继续读取本地 PNG 后按新版卡片轮廓裁切。
- 底图上方不得叠加半透明柔光、灰色遮罩或其他覆盖层。

如果只有一张名片：
- 页面不显示底部翻页胶囊提示。
- 名片本身不需要左右分页提示。

如果有多张名片：
- 多张名片横向排布。
- 用户可左右滑动切换，交互类似手机翻页。
- 只有名片区域参与滑动，页面其他元素不滑动。
- 多张名片需要循环滑动。
- 滑动过程中相邻两张名片的水平间距为 32pt。
- 名片应跟随手势立即移动，降低松手后的切换延迟。
- 左滑切换下一张时，下一张名片从右侧滑入，当前名片从左侧滑出；右滑切换上一张时，上一张名片从左侧滑入，当前名片从右侧滑出。两张名片的滑动动画曲线和时长保持一致。
- 底部胶囊型翻页提示跟随当前名片变化。
- 每增加 1 张名片，胶囊宽度增加 20pt，并增加 1 个直径 8pt 的圆点。
- 每减少 1 张名片则反向减少。
- Figma `7575:3207` 中分页胶囊为 x=171, y=716, h=20；当前实现按最新指令整体下移 34pt，即顶部 y=750、中心 y=760；宽度为 `名片数 * 20`。
- 胶囊背景色为 `#D9D9D9`，圆点直径 8pt，当前圆点 `#FFFFFF`，非当前圆点 `#EDEDED`，圆点间距 12pt。
- 分页胶囊不使用阴影。

## 5. 名片生成与布局规则

> 本节是 Carda 当前最重要的实现约束。名片生成必须按这里执行。

### 5.1 基准画板与名片尺寸

基准 frame：`9978:2322`（新版带右侧按钮名片）

页面画板：
- w=402
- h=874

基准名片：
- x=16
- y=157
- w=370
- h=222
- cornerRadius=26（除裁切槽顶部两个受宽度限制的倒角外）；名片外框与右下裁切槽均使用普通 `.circular` / 圆弧圆角，不使用 Corner smoothing、`.continuous` 或平滑圆角
- 背景：白色卡片 + `Card photo/Group 42.png` 底图图片；底图继续使用本地 PNG，不重做 Figma 光斑；不得叠加半透明柔光、灰色遮罩或其他覆盖层
- 默认阴影：Figma `9978:2322` 为 `0 0 4 rgba(0,0,0,0.25)`；但“我的名片”页面按 Figma `10063:3262` 去掉名片本体外框阴影
- 新版名片右下角有按钮槽裁切：当前实现的基准裁切槽相对名片为 x=329, y=103, w=41, h=119；动态高度下裁切槽跟随右侧按钮列向上扩展并向下贴齐名片底边
- 裁切槽左边相对上一版向右移动 7pt；槽顶靠近最上方按钮的两个倒角若与槽宽冲突，半径自适应降到槽宽的 1/2（当前最大为 20.5pt），其余倒角在原本 24pt 基础上增加 2pt，当前为 26pt；若多按钮场景因槽高不足产生倒角冲突，实现应优先略微上提槽顶，而不是压小其他倒角；但当右侧只有 1 个按钮 / 1 条信息时，裁切槽不使用该上提保护，槽顶需相对上一保护结果向下移动 5.5pt
- 卡片绘制顺序必须为：先按当前动态高度裁切 PNG 底图，再切除右下角按钮槽，再对最终轮廓统一倒角，最后只对最终轮廓绘制阴影；不得让被切除区域继续保留阴影或方角

说明：
- 用户描述中提到“370px * 222px”。Figma 坐标显示 370 为横向宽度，222 为纵向短边。
- 后续动态增加的是 222 这条短边，也就是实现中的名片高度。
- 当前名片底图不再由 SwiftUI 自绘图形生成，必须直接读取项目本地 `Card photo/Group 42.png`。
- 底图图片原始尺寸为 370 x 300；渲染时按名片区域居中 `scaledToFill`，根据当前名片高度上下裁切。

### 5.7 名片动态高度规则

基准状态：
- 最多 3 段单行信息时，名片高度为 222pt。

动态增长：
- 如果信息字段超过 3 段，每多 1 段，名片高度增加 24pt。
- 如果某一段信息发生换行，每多 1 行，名片高度额外增加 20pt。
- 信息字段文本单元本身也必须按每多 1 行增加 20pt 排布，确保总高度增长后信息模块仍按底部锚定。
- 横向宽度 370pt 保持不变。
- 名片左上角在页面中的 x/y 不应因高度变化而漂移，除非后续用户明确要求。
- 公司组与头像保持顶部锁定。
- 姓名组保持竖向居中。
- 信息模块保持底部锁定。
- 右侧操作按钮列保持底部锁定；动态高度公式不因按钮列额外增加高度。

建议公式：

```text
cardHeight = 222
  + max(0, visibleInfoFieldCount - 3) * 24
  + wrappedExtraLineCount * 20
```

其中：
- `visibleInfoFieldCount` 为右下角实际显示的信息字段数量。
- `wrappedExtraLineCount` 为所有字段换行后超出第一行的总行数。

### 5.8 保存为图片

保存名片为图片时：
- 输出格式为 `.png`。
- 输出内容只包含名片本体，不包含页面背景、导航栏、弹窗或遮罩。
- 输出应使用当前名片真实动态高度。
- 头像、公司 logo、底图和右侧字段操作按钮都必须被渲染进图片。

## 8. 名片交换

Carda 的主要交换方式为 App 内顶部碰一碰式近距离交换，目标是第三方 App 可实现的近距离体验，不调用 AirDrop、NameDrop 或 NFC。

交换对象永远是“我的名片”页面当前正在展示的那张名片。

### 8.1 当前启用边界

- 只有双方都打开 Carda，并停留在“我的名片”页面时才启用交换能力。
- 当前页面必须存在可交换的当前名片。
- 创建/编辑页、添加名片弹窗或离开“我的名片”页面时，必须停止 Multipeer Connectivity 与 Nearby Interaction session。
- 收到的名片保存为 `ownerKind = .received`，进入名片夹数据源。
- 当前阶段按用户要求只实现功能，不实现交换成功动画；允许保留短文本提示和震动反馈。

### 8.2 双方交换技术方案

- 使用 Multipeer Connectivity 自动发现附近同款 App 用户、建立连接，并传输 Nearby Interaction discovery token 与名片数据。
- 使用 Nearby Interaction 启动 `NISession`，读取对方设备的 `distance` / `direction`。
- Multipeer service type 当前为 `carda-ex`；`Info.plist` 必须配置 Bonjour service `_carda-ex._tcp` 和本地网络用途说明。
- 连接建立后双方交换 discovery token，随后用收到的 token 启动 `NINearbyPeerConfiguration`。

### 8.3 顶部碰一碰触发规则

- 触发目标是两台 iPhone 顶部互相贴住，而不是泛化的近距离路过。
- 当前阈值：`distance <= 0.12m`。
- 需要信号稳定持续至少 0.5 秒。
- 如果 `direction` 可用，要求对方位于手机顶部方向，当前实现以 `direction.y >= 0.45` 作为轻量方向门槛。
- 如果 `direction` 不可用，则仅按距离和稳定时间判断。
- 同一 peer 的发送和接收各自设置 5 秒冷却，防止重复触发。

### 8.4 尚未实现 / 待验证

- 需要两台支持 Nearby Interaction 精确测距的 iPhone 做端到端真机验证；模拟器不能验证 UWB distance / direction。
- 一方未打开 App、后台接收、未安装 App 投递、单向递交和交换成功动画当前均未实现。
- 后续不得用 AirDrop、NameDrop 或 NFC 作为该近距离交换能力的实现方案，除非用户明确改变边界。

## 10. 数据模型方向

正式实现前需要建立名片数据模型。初步方向如下，写代码前仍需与用户确认字段命名和持久化策略。

核心对象：
- `BusinessCard`：一张名片。
- `CardField`：一段可变信息。
- `BusinessCardList`：名片夹中的一个自定义列表。
- `CardAsset`：头像、公司 logo 等用户上传图片资源；名片底图为项目固定资源 `Card photo/Group 42.png`，不进入名片数据模型。

名片字段建议：
- `id`
- `ownerKind`：我的名片 / 收到的名片。
- `name`
- `phoneticName`
- `position`
- `organizationName`
- `avatarImage`
- `companyLogoImage`
- `cardListID`：收到的名片当前唯一归属列表；为空或指向无效列表时进入 `未分类`。
- `fields`
- `createdAt`
- `updatedAt`
- `receivedAt`

信息字段建议：
- `id`
- `type`：手机 / 邮箱 / 地址 / 链接
- `value`
- `sortOrder`

公司 logo 使用 `companyLogoImage` / `companyLogoImageData` 存储，不再作为动态信息字段的可选类型。

列表字段：
- `id`
- `name`
- `sortOrder`
- `createdAt`
- `updatedAt`

## 11. AI 开发规则

AI 在修改 Carda 代码或文档时必须：

1. 先阅读 `AGENTS.md`、`contributing_ai.md`、`README.md`、`Designsystem.md` 和 `ROADMAP.md`。
2. 以 Figma Page 1 与用户描述为主要依据。
3. 遇到任何不明确、不确定或存在多种实现方向的地方，先询问用户。
4. 不自行补全未设计页面，例如名片底图切换页。
5. 不自行更改名片生成规则。
6. 修改名片布局、动态高度、保存图片或交换逻辑前，必须说明影响范围。
7. 不覆盖用户已有改动，不删除未经确认的内容。
8. 新增设计、架构、功能决策后，同步更新相关文档。
